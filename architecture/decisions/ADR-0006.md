# ADR-0006: Event-Command-Query Separation

**Status:** Accepted | **Date:** 2026-07-17 | **Author:** Principal Architect

## Context
Domain operations include state-changing actions (commands), information retrieval (queries), and notifications (events). These must be separated for clarity and future CQRS/event sourcing support.

## Decision
Separate commands, queries, and events into distinct catalogs:
- **Commands**: State-changing operations (CreateCharacter, UpdateBook)
- **Queries**: Read-only data retrieval (FindCharacterById, ListBooksBySeries)
- **Events**: Domain event notifications (CharacterCreated, BookPublished)

## Alternatives Considered
1. Combined CRUD — Simple but loses intent; rejected
2. Only commands — Queries less structured; rejected
3. Full CQRS/Event Sourcing — Over-engineered for Phase 1; deferred to Phase 9

## Rationale
- Clear separation of writes, reads, and notifications
- Enables future CQRS without refactoring
- Commands and queries can be independently validated
- Events support cross-context communication (narrative → timeline)

## Consequences
- 60+ commands defined across 9 categories
- 50+ queries defined including cross-domain queries
- 50+ events defined across narrative, character, world, timeline, etc.
- Event-driven architecture pattern established

## Future Impact
- Phase 6 Narrative Tools: Commands map directly to tool operations
- Phase 9 Database: CQRS separation already designed in
- Phase 10 Collaboration: Events become audit trail and notification system
