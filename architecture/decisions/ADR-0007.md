# ADR-0007: Configuration Strategy

**Status:** Accepted | **Date:** 2026-07-17 | **Author:** Principal Architect

## Context
The project needs configuration values (project name, AI rules, naming rules, ID schema) that are referenced by standards documents, validation scripts, and AI prompts.

## Decision
Store configuration in JSON files in `config/` with:
- One file per concern (project, settings, metadata, version, AI rules, writing rules, naming rules, ID rules)
- Empty `{}` as placeholders (populated during Phase 2)
- Documentation-driven: standards documents describe what config values should be
- Config values override defaults but cannot change core standards

## Alternatives Considered
1. Environment variables — Platform-dependent, not commit-friendly
2. Single monolithic config file — Becomes unwieldy
3. YAML — Less supported than JSON in AI ecosystems
4. No config — Hardcoded values; rejected — not maintainable

## Rationale
- JSON is universally supported (AI, Python, Node, browsers)
- File-based config is Git-trackable
- Separate files per concern prevent conflicts
- Empty placeholders signal Phase 2 work items

## Consequences
- 8 JSON files, all currently empty (`{}`)
- Config population is a pre-Phase-2 requirement
- Config values must be documented in corresponding standards
- Config changes must be validated against frozen architecture

## Future Impact
- Phase 2 Schema: Config values used in template generation
- Phase 3 Validation: Config values used as validation parameters
- Phase 4 Prompts: AI rules config constrains AI behavior
- Phase 10 Collaboration: Multi-tenancy config added
