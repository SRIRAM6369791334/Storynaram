# ADR-0004: Entity Ownership and Lifecycle

**Status:** Accepted | **Date:** 2026-07-17 | **Author:** DDD Architect

## Context
Entities have containment relationships (Book contains Chapters) and lifecycle states (Draft → Review → Approved). These must be formally defined.

## Decision
- Separate ownership hierarchy from entity identity
- Ownership is containment (Book ◆— Chapter), not inheritance
- Every entity has a defined lifecycle state machine
- Transitions are validated against allowed paths
- Lifecycle events are logged

## Alternatives Considered
1. No formal lifecycle — Entities always editable; rejected — canon integrity requires lifecycle
2. Single lifecycle for all entities — Different entities have different workflows; rejected
3. Ownership embedded in entity ID — Too rigid; rejected

## Rationale
- Story entities have different workflows (Books need publishing pipeline; Characters need canon approval)
- Canon integrity requires locking
- Ownership clarifies deletion cascades

## Consequences
- Entity-specific lifecycle state machines
- Transition validation required for all mutations
- Ownership cascade rules for delete operations
- Status change events for cross-context notifications

## Future Impact
- Phase 3 Validation: Lifecycle transition validation
- Phase 5 Canon: Locked states integrated with canon engine
- Phase 6 Narrative Tools: Workflow automation based on lifecycles
