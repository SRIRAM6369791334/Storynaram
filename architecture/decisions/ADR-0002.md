# ADR-0002: Domain Model Separation

**Status:** Accepted | **Date:** 2026-07-17 | **Author:** DDD Architect

## Context
The domain model must be expressed independently of implementation technology, persistence strategy, and AI infrastructure.

## Decision
Implement a pure DDD domain model with:
- Explicit bounded contexts (7 total)
- Aggregate roots for transactional consistency (7 total)
- Value objects for immutable properties (20+ types)
- Domain services for cross-entity operations
- Repository interfaces (not implementations)
- Domain events for cross-context communication

## Alternatives Considered
1. Anemic domain model — Entities as pure data containers; rejected — violates DDD
2. Active Record pattern — Rejected — couples domain to persistence
3. No domain model — Rapid initial development but unsustainable at scale

## Rationale
- DDD is the industry standard for complex domains
- Story creation is inherently complex with deep entity relationships
- Enables domain experts to understand and validate the model
- Persistence-agnostic design enables future database migration

## Consequences
- 17 domain subdirectories
- 86 entity types with formal definitions
- Clear separation between domain (what) and infrastructure (how)
- Requires schema/template layer (Phase 2) to bridge domain and persistence

## Future Impact
- Phase 9 Database: Repositories get implementations; domain unchanged
- Phase 3 Validation: Validators enforce domain rules
