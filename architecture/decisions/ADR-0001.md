# ADR-0001: Folder Structure and Organization

**Status:** Accepted | **Date:** 2026-07-17 | **Author:** Principal Architect

## Context
The project needs a consistent, scalable folder structure that supports Clean Architecture, DDD, AI integration, and long-term maintainability.

## Decision
Adopt a layered directory structure:
- `config/` — Runtime configuration (JSON files)
- `core/` — Immutable standards, contracts, enums, types
- `domain/` — Domain model with DDD tactical patterns
- `ai/` — AI Knowledge Architecture blueprint
- `scripts/` — Automation and tooling
- Each layer depends only on layers below it

## Alternatives Considered
1. Flat structure — Too chaotic at scale
2. Feature-based grouping — Cross-cutting concerns duplicated
3. Monolithic single directory — Unmanageable for 86 entity types

## Rationale
- Clean Architecture dependency direction (inward)
- DDD tactical patterns need explicit directories
- AI architecture is a separate concern from domain
- Configuration must be immediately discoverable

## Consequences
- 89 directories for Phase 1; will grow with content
- Clear separation of concerns
- Easy to enforce dependency rules
- Migration-friendly (each layer independently replaceable)

## Future Impact
- Phase 9 Database: Database layer would sit at same level as file-based storage
- Phase 10 Collaboration: Auth and multi-tenancy would add new top-level directories
