# ADR-0008: Documentation-First Approach

**Status:** Accepted | **Date:** 2026-07-17 | **Author:** Principal Architect

## Context
The project has no code yet — only documentation. This is intentional: architecture must be frozen before implementation begins.

## Decision
Adopt a documentation-first approach:
- All architecture is documented in Markdown
- All diagrams are Mermaid (text-based, version-controllable)
- Standards are written documents before machine-readable schemas
- README.md in every directory
- Documentation is the source of truth

## Alternatives Considered
1. Code-first — Rapid prototyping; rejected — leads to architecture drift
2. UML-first — Tool-dependent, not easily diff-able; rejected
3. Wiki — Not in-repo; rejected — documentation drifts from code

## Rationale
- Documentation is version-controllable in Git
- Mermaid diagrams render natively on GitHub and VS Code
- Non-technical domain experts can review and validate
- Machine-readable formats (JSON Schema) are generated FROM documentation

## Consequences
- 184 files, 89 directories — all documentation, no code
- Markdown with Mermaid is the universal format
- Future code must conform to documented architecture
- Architecture review is document review, not code review

## Future Impact
- All future phases start with documentation updates
- Phase 2 Schema: JSON Schema generated from contract documentation
- Phase 5 Canon: Canon change workflow starts with document RFC
- Long-term: Documentation is the system specification
