# Reference Standard

## Purpose
Defines the cross-entity reference system for Storynaram. References are how entities link to each other without duplicating data.

## Reference Format
References are always the target entity's full ID as a string:

`json
"hero_000001"
`

Or in relationship blocks:
`json
{
  "relationships": {
    "parent": "book_000001",
    "children": ["chapter_000001", "chapter_000002"],
    "related": ["location_000001", "event_000042"]
  }
}
`

## Reference Types

### Direct Reference
A single entity ID pointing to another entity:
`json
"currentLocation": "city_000001"
`

### Array Reference
Multiple entity IDs in an array:
`json
"characters": ["hero_000001", "heroine_000002", "villain_000003"]
`

### Typed Reference
A reference with type metadata:
`json
{
  "id": "hero_000001",
  "type": "character",
  "role": "protagonist"
}
`

## Reference Fields in Metadata

Every relationship block uses these field types:
| Field | Type | Description |
|-------|------|-------------|
| parent | string or null | Single parent entity ID |
| children | array of strings | Child entity IDs |
| elated | array of strings | Related entity IDs |
| eferences | array of strings | Source/reference entity IDs |

## Bidirectional Reference Rule
When entity A references entity B, entity B should include an inverse reference to entity A when feasible. This ensures bidirectional traversal without requiring full graph scans.

## Reference Validation Rules
1. Every reference must point to an existing entity ID
2. References must match the expected type (no character reference in a location field)
3. Circular references are allowed but must be intentional
4. Orphaned references (pointing to deleted/archived entities) are not allowed
5. Reference arrays cannot contain duplicates

# Naming Standard

## Purpose
Defines naming conventions for all named elements in Storynaram â€” entities, files, directories, fields, variables, and display names.

## Entity Display Names
Entity display names (the 
ame field in JSON) follow these rules:
- **Title Case** for characters, locations, organizations: "King Aldric the Brave", "The Crystal Palace"
- **Title Case** for books, chapters, scenes: "The Fellowship of the Ring", "Chapter One: A Long-Expected Party"
- **Proper capitalization** â€” always capitalize proper nouns
- **No ALL CAPS** except acronyms: "MINASA Morgul" (no), "Minas Morgul" (yes)
- **Maximum length**: 120 characters for display names

## Character Names
- First name + Last name format where applicable
- Titles included in display name: "King Aldric III"
- Nicknames in quotes: "Aldric \"the Brave\" Stormwind"
- Species/race identifiers when relevant: "Aldric Stormwind (Human)"

## Location Names
- Geographic name format: "The Crimson Mountains", "Lake Serenity"
- Constructed names follow conlang rules from languages/
- Real-world inspired names are permitted
- Avoid generic names without context: "Big Forest" (no), "The Whispering Woods" (yes)

## Item Names
- Descriptive names: "Blade of a Thousand Sorrows", "Elixir of Eternal Vigor"
- Quality prefix optional: "Masterwork Steel Longsword"
- Quantity for consumables: "Healing Potion (x3)"

## File Naming
- Entity files: {prefix}_{sequence}.json â€” hero_000001.json
- Schema files: {EntityType}.schema.json â€” Character.schema.json
- Template files: {entity_type}_template.json â€” character_template.json
- Standards: UPPER_SNAKE_CASE.md â€” JSON_STANDARD.md
- Documentation: UPPER_SNAKE_CASE.md for root docs, README.md for dir docs
- Scripts: {action}_{target}.ps1 â€” alidate_characters.ps1
- Prompts: {category}_{purpose}.md â€” character_generation_prompt.md

## Field Naming (JSON keys)
- **camelCase** â€” characterName, currentLocation, isAlive
- Acronyms: iRules (not AIRules), jsonSchema (not JSONSchema)
- Avoid abbreviations: description (not desc), organization (not org)
- Common abbreviations allowed: id, max, min, vg, config, ef
- Boolean fields: isActive, hasChildren, canFly, shouldNotify

## Variable Naming (Scripts)
- PowerShell: $camelCase with PascalCase for functions
- Python: snake_case for variables and functions
- JavaScript: camelCase for variables, PascalCase for classes
- Constants: UPPER_SNAKE_CASE across all languages

## Directory Naming
- **snake_case** â€” all lowercase with underscores
- Plural for entity collections: characters/, items/, spells/
- Singular for functional directories: config/, core/, memory/
- Numbers zero-padded: ook_001/, chapter_01/

## Product/Brand Naming
- The project is named Storynaram â€” always capitalized as such
- Never storynaram, STORYNARAM, or Story-naram
- The abbreviation SN may be used in internal contexts
- Storynaram is one word â€” not Story Naram

# Metadata Standard

## Purpose
Defines the standard metadata block that must be present in every entity JSON file in Storynaram.

## Standard Metadata Block
Every entity JSON must include a metadata object with these fields:

`json
{
  "metadata": {
    "id": "hero_000001",
    "name": "King Aldric Stormwind",
    "description": "The rightful heir to the throne of Eldoria",
    "version": 1,
    "status": "draft",
    "createdAt": "2026-07-17T12:00:00Z",
    "updatedAt": "2026-07-17T12:00:00Z",
    "createdBy": "author",
    "updatedBy": "author",
    "tags": ["protagonist", "human", "royalty"],
    "references": ["ref_000001", "ref_000002"],
    "notes": "Original concept from outline v1"
  }
}

## Field Specifications

### id (required)
- Type: string
- Format: {prefix}_{sequence} per ID_STANDARD
- Immutable once assigned
- Globally unique

### 
ame (required)
- Type: string
- Format: Title Case per NAMING_STANDARD
- Maximum 120 characters
- Should be human-readable and descriptive

### description (recommended)
- Type: string
- Format: Sentence case, complete sentences
- Maximum 2000 characters
- Brief summary of the entity's purpose or nature

### ersion (required)
- Type: integer
- Starts at 1
- Incremented on each modification
- See VERSION_STANDARD

### status (required)
- Type: string
- Allowed values: draft, eview, evised, inal, published, rchived, deprecated
- See enums/STATUS.md

### createdAt (required)
- Type: string (ISO 8601)
- Format: YYYY-MM-DDTHH:mm:ssZ
- Set once at creation, never modified

### updatedAt (required)
- Type: string (ISO 8601)
- Format: YYYY-MM-DDTHH:mm:ssZ
- Updated on every modification

### createdBy (recommended)
- Type: string
- Values: uthor, i, system, or a username
- Identifies the creator of the entity

### updatedBy (recommended)
- Type: string
- Same format as createdBy
- Updated on every modification

### 	ags (recommended)

### eferences (optional)
